Projeto: Sistema de Gerenciamento de Academias

Nome do Sistema: JyM

1.0 INTRODUÇÃO

A gestão eficiente de academias tornou-se um fator crucial para o sucesso e a sustentabilidade desses estabelecimentos em um mercado cada vez mais competitivo. Com o aumento da procura por atividades físicas e bem-estar, os gestores enfrentam desafios relacionados ao controle de clientes, cobranças de mensalidades, controle de acesso e organização de dados. Muitos desses processos ainda são realizados manualmente ou com ferramentas genéricas, o que pode comprometer a produtividade, a segurança das informações e a experiência dos usuários.

Diante desse cenário, a adoção de sistemas informatizados específicos para academias tem se mostrado essencial. Esses sistemas proporcionam maior controle, agilidade nas rotinas administrativas e precisão no acompanhamento financeiro. Além disso, contribuem para a fidelização dos clientes ao oferecer uma experiência mais organizada e segura. Nesse contexto, surge o sistema "JyM", desenvolvido como uma aplicação web para suprir essas necessidades de maneira prática, moderna e acessível.

O sistema "JyM" é uma aplicação web desenvolvida para facilitar o gerenciamento de academias. Seu objetivo é oferecer uma solução prática e eficiente para o controle de clientes, mensalidades e autorização de entrada, além de proporcionar uma interface de acesso seguro por meio de login e cadastro. O sistema será implementado utilizando tecnologias web, garantindo a usabilidade.

1.1 Objetivo do Sistema

O JyM tem como propósito principal otimizar a gestão de academias, permitindo:

  • Gestão Centralizada e Multi-Academia: Cadastrar e gerenciar múltiplas unidades de academias a partir de um único painel, com segregação de dados entre elas.
  • Controle de Acesso Avançado: Autorizar a entrada de clientes de forma segura e eficiente, utilizando reconhecimento facial como método principal e um sistema de fallback (Código de acesso) para contingências. Verificar acesso utilizando status do cliente ‘Ativo’; negar quando diferente de `Ativo`.
  • Gestão Completa de Clientes e Planos: Administrar o ciclo de vida dos clientes, desde o cadastro e captura facial até a gestão de seus planos de assinatura, renovações e histórico de pagamentos.
  • Administração de Usuários com Níveis de Acesso: Registrar e autenticar usuários do sistema (administradores e funcionários), controlando suas permissões de acesso às funcionalidades.
  • Gestão Financeira e de Vendas: Controlar mensalidades, registrar o status de pagamento e gerenciar a venda de produtos, incluindo funcionalidades de controle de estoque e estorno de vendas.
  • Monitoramento e Análise em Tempo Real: Oferecer um dashboard com métricas e relatórios em tempo real sobre as operações da academia, auxiliando na tomada de decisões estratégicas.

2.0 DOCUMENTO DE REQUISITOS

Fundamentação Teórica: Documento de Requisitos no Sistema JyM

O documento de requisitos, conforme Sommerville (2011), é um registro formal que detalha os requisitos funcionais, não funcionais e regras de negócio de um sistema, funcionando como um contrato entre desenvolvedores e clientes. Serve para capturar e organizar expectativas, orientar o desenvolvimento e validar o sistema. No "JyM", o documento especificou funcionalidades como "Fazer Login" e "Gerenciar Mensalidades", além de restrições como segurança e desempenho, guiando a modelagem UML (casos de uso, diagramas de classes e sequência) e a implementação, assegurando que o sistema atendesse às necessidades dos stakeholders.

2.1. Requisitos Funcionais

Os requisitos funcionais descrevem as funcionalidades que o sistema deve oferecer:

2.1.1 Autenticação e Autorização

  • RF01: O sistema deve possuir uma tela de login segura com campos para usuário e senha.
  • RF02: O sistema deve validar as credenciais do usuário e, em caso de sucesso, direcioná-lo ao dashboard principal, aplicando o contexto da sua academia.
  • RF03: O sistema deve exibir uma mensagem de erro clara caso as credenciais de login estejam incorretas.
  • RF04: O sistema deve permitir a administradores o cadastro de novos usuários, associando-os a uma academia e definindo seu nível de acesso.
  • RF05: O cadastro de usuários deve validar a unicidade de usuário e e-mail (quando informado)  para evitar duplicidade.

2.1.2 Gerenciamento de Clientes

  • RF06: O sistema deve permitir o cadastro completo de clientes, incluindo dados pessoais (nome, CPF, data de nascimento), informações de contato e a captura de imagem para o reconhecimento facial.
  • RF07: O sistema deve apresentar uma listagem paginada de todos os clientes, com ferramentas de busca e filtros avançados (por nome, CPF, status, plano).
  • RF08: O sistema deve permitir a edição dos dados de um cliente.
  • RF09: O status do cliente deve ser claramente visível e estar diretamente ligado à adimplência de sua mensalidade.
  • RF10: O sistema deve implementar exclusão lógica (soft delete) para clientes, permitindo restauração posterior.
  • RF11: O sistema deve permitir a visualização de clientes excluídos (apenas para administradores).
  • RF12: O sistema deve permitir restauração de clientes excluídos logicamente (apenas para administradores).
  • RF13: O sistema deve permitir exclusão física permanente de clientes (apenas para administradores). [a]

2.1.3 Gerenciamento de Mensalidades e Planos

  • RF14: O sistema deve permitir o registro e o acompanhamento de mensalidades associadas a cada cliente, refletindo o plano de assinatura contratado.
  • RF15: O sistema deve exibir listas de mensalidades pendentes e vencidas, com notificações visuais claras (ex: cores) para fácil identificação.
  • RF16: O sistema deve permitir que um funcionário registre o pagamento de uma mensalidade, alterando seu status de "pendente" para "pago".
  • RF17: O sistema deve registrar a data e forma de pagamento ao processar uma mensalidade.
  • RF18: O sistema deve atualizar automaticamente o status do cliente após pagamento de mensalidades.[b]
  • RF19: O sistema deve permitir o cadastro, edição e exclusão de diferentes planos de assinatura, definindo nome, descrição, valor e duração.
  • RF20: O sistema deve permitir renovação de planos de clientes existentes.

2.1.4 Controle de Acesso (Modo Kiosk)

  • RF21: O sistema deve operar um modo Kiosk dedicado para o controle de acesso, utilizando reconhecimento facial como método primário de autenticação.
  • RF22: Como método de fallback, o Kiosk deve permitir a autenticação via código de acesso numérico de 6 dígitos (gerado automaticamente no momento do cadastro do cliente).
  • RF23: Ao identificar um cliente, o sistema deve verificar em tempo real o status de sua mensalidade e liberar ou negar o acesso.
  • RF24: O sistema deve registrar todas as entradas de clientes com método de autenticação utilizado e data/hora.
  • RF25: O sistema deve permitir acesso para clientes com status "Ativo", negando para outros status.[c]

2.1.5 Gestão Multi-Academia

  • RF26: O sistema deve ser capaz de gerenciar múltiplas academias de forma centralizada.
  • RF27: Usuários com perfil de administrador devem poder alternar entre as academias que gerenciam para visualizar e editar dados específicos de cada unidade.
  • RF28: Todos os dados (clientes, planos, vendas, produtos, entradas, fornecedores, compras, equipamentos, materiais) devem ser estritamente segregados por academia.
  • RF29: O sistema deve aplicar filtros automáticos por academia em todas as consultas para funcionários.
  • RF30: Administradores devem poder visualizar e gerenciar dados de múltiplas academias através de seletor de contexto.

2.1.6 Gerenciamento de Produtos e Vendas

  • RF31: O sistema deve permitir o cadastro completo de produtos, incluindo nome, categoria, marca, fornecedor, preço de venda, preço de custo, estoque mínimo, estoque, descrição e imagem.
  • RF32: O sistema deve permitir o cadastro de categorias de produtos vinculadas à academia.
  • RF33: O sistema deve permitir o cadastro de marcas de produtos com informações complementares (país de origem, site).
  • RF34: O sistema deve calcular e armazenar o custo médio ponderado dos produtos automaticamente.[d]
  • RF35: O sistema deve controlar estoque mínimo e alertar sobre produtos com necessidade de reposição.
  • RF36: O sistema deve possuir uma interface para registrar vendas de produtos, associando-as a um cliente (opcional configurável) e ao funcionário que realizou a venda.
  • RF37: O sistema deve calcular automaticamente o valor total da venda e permitir a seleção da forma de pagamento (Dinheiro, Cartão de Crédito, Cartão de Débito, PIX, Boleto).
  • RF38: Após a conclusão de uma venda, o estoque dos produtos correspondentes deve ser atualizado automaticamente.
  • RF39: O sistema deve registrar todas as movimentações de estoque com fornecedor, quantidade, status e valor.
  • RF40: O sistema deve permitir o estorno de uma venda, revertendo a operação e ajustando o estoque de volta.
  • RF41: O sistema deve manter um histórico detalhado de todas as vendas realizadas.

2.1.7 Gestão de Fornecedores e Compras

  • RF42: O sistema deve permitir o cadastro completo de fornecedores incluindo razão social, CNPJ/CPF, inscrição estadual, contato, telefone, email, endereço e condição de pagamento padrão[e].
  • RF43: O sistema deve permitir ativar/desativar fornecedores mantendo o histórico.
  • RF44: O sistema deve permitir a criação de pedidos de compra vinculados a fornecedores.
  • RF45: O sistema deve permitir adicionar múltiplos itens em uma compra com quantidade e preço unitário.
  • RF46: O sistema deve calcular automaticamente o valor total da compra incluindo produtos, frete, descontos e impostos.[f]
  • RF47: O sistema deve criar automaticamente uma conta a pagar ao registrar uma compra.
  • RF48: O sistema deve permitir o recebimento de compras, atualizando status, estoque e custo médio[g] dos produtos.
  • RF49: Ao receber uma compra, o sistema deve registrar a movimentação de entrada no estoque.
  • RF50: O sistema deve impedir recebimento de compras que não estejam com status "aberta".

2.1.8 Gestão Financeira

  • RF51: O sistema deve permitir o cadastro de categorias para contas a pagar.[h]
  • RF52: O sistema deve registrar automaticamente as contas a pagar ao receber compras.
  • RF53: O sistema deve permitir listagem e filtros de contas a pagar por status e período.
  • RF54: O sistema deve permitir o registro de pagamento de contas a pagar com data e forma de pagamento.
  • RF55: O sistema deve manter estrutura para contas a receber (preparada para futuras integrações).
  • RF56: O sistema deve vincular contas a pagar/receber às suas origens (compra, venda, mensalidade).

2.1.9 Gestão de Equipamentos e Manutenção

  • RF57: O sistema deve permitir o cadastro de equipamentos com descrição, fabricante, modelo, número de série, data de aquisição, valor e fim de garantia.
  • RF58: O sistema deve controlar o status dos equipamentos: Ativo, Em Manutenção ou Desativado.
  • RF59: O sistema deve permitir associar centro de custo aos equipamentos.
  • RF60: O sistema deve permitir o registro de manutenções de equipamentos incluindo data, problema, serviço realizado, custo e responsável.
  • RF61: O sistema deve manter histórico completo de manutenções por equipamento.

2.1.10 Gestão de Materiais de Consumo

  • RF62: O sistema deve permitir o cadastro de materiais de consumo com descrição, estoque, unidade de medida e estoque mínimo.
  • RF63: O sistema deve permitir o controle de estoque de materiais separado do estoque de produtos.
  • RF64: O sistema deve permitir criação de requisições de materiais.
  • RF65: O sistema deve alertar quando os materiais atingirem o estoque mínimo.[i]

2.1.11 Relatórios e Análises

  • RF66: O sistema deve gerar relatório de compras com filtros por período e fornecedor.
  • RF67: O sistema deve gerar relatório de margem de produtos calculando lucro sobre vendas.
  • RF68: O sistema deve gerar relatório de ruptura de estoque listando produtos abaixo do estoque mínimo.
  • RF69: O sistema deve gerar relatório de faturamento e lucro por período.
  • RF70: O sistema deve gerar relatório de gastos consolidando contas a pagar.
  • RF71: O sistema deve gerar relatório de inadimplência listando clientes com mensalidades vencidas.[j]
  • RF72: O sistema deve gerar relatório de frequência de clientes baseado em entradas.
  • RF73: O sistema deve gerar relatório de vendas com análise por período e produto.
  • RF74: O sistema deve gerar relatório de desempenho por funcionário.[k]
  • RF75: Todos os relatórios devem permitir exportação em formato PDF.[l]

2.10. Requisitos Não Funcionais

Os requisitos não funcionais definem as características de qualidade e restrições do sistema:

  • RNF01 (Usabilidade): A interface deve ser intuitiva, limpa e organizada, utilizando os princípios de design do Tailwind CSS e a reatividade do Alpine.js para minimizar a curva de aprendizado e facilitar o uso diário.
  • RNF02 (Performance): O sistema deve ter tempos de resposta rápidos. A performance é otimizada através da compilação de assets com Vite, indexação eficiente do banco de dados e uso de cache quando aplicável.
  • RNF03 (Segurança): O sistema deve garantir a segurança dos dados e do acesso. As senhas são protegidas com hashing Bcrypt. O framework Laravel fornece proteção nativa contra vulnerabilidades comuns como CSRF e SQL Injection.
  • RNF04 (Escalabilidade): A arquitetura do sistema e do banco de dados deve ser capaz de suportar o crescimento no número de academias, clientes e transações sem degradação significativa de performance.

2.11. Regras de Negócio

  • RN01: O acesso de um cliente à academia será concedido se ele possuir status "Ativo". Será negado para outros status.
  • RN02: Mensalidades com status "Pendente" e data de vencimento passada são consideradas atrasadas. Um job automático atualiza clientes para "Inadimplente" quando há pendências vencidas.
  • RN03: Apenas usuários autenticados podem acessar as funcionalidades de gerenciamento do sistema.
  • RN04: O sistema diferencia dois níveis de acesso: Administrador (acesso total, incluindo configurações de múltiplas academias) e Funcionário (acesso operacional restrito à sua academia designada).
  • RN05: O estorno de uma venda implica na reversão da transação financeira e no retorno dos itens vendidos ao estoque.
  • RN06: O custo médio ponderado é calculado pela fórmula: custoMedio = ((custoAtual × estoqueAtual) + (preçoNovo × quantidadeNova)) / estoqueNovo quando o estoque resultante for maior que zero.[m]
  • RN07: Compras só podem ser recebidas se estiverem com status "aberta".
  • RN08: Ao receber uma compra, o estoque é atualizado automaticamente, o custo médio é recalculado e o status da compra muda para "recebida".
  • RN10: O sistema utiliza exclusão lógica (soft delete) para clientes, permitindo restauração posterior.
  • RN11: Todas as movimentações de estoque são registradas para rastreabilidade completa no banco de dados.
  • RN12: Equipamentos possuem três status possíveis: Ativo, Em Manutenção ou Desativado.
  • RN13: Produtos com estoque menor ou igual ao estoque mínimo são sinalizados no relatório de ruptura para reposição.
  • RN14: Clientes excluídos logicamente podem ser restaurados apenas por administradores.
  • RN15: Exclusão física permanente só é permitida para clientes sem vínculos com mensalidades, entradas ou vendas.[n]
  • RN16: Cada compra gera automaticamente uma conta a pagar com vencimento padrão em 30 dias (se não especificado).

2.13. Dentro do Escopo

  • Gestão Completa de Múltiplas Academias: Cadastro e gerenciamento de unidades, com segregação de dados.
  • Gestão de Usuários e Permissões: Cadastro de administradores e funcionários com diferentes níveis de acesso.

  • Gestão de Produtos e Estoque: Cadastro de produtos, controle de quantidade em estoque e atualizações automáticas.
  • Registro de Vendas e Estorno: Funcionalidade completa para registrar vendas de produtos e realizar estornos, com os devidos ajustes de estoque.
  • Dashboard e Relatórios: Painel com métricas em tempo real e geração de relatórios avançados sobre faturamento, retenção e outros indicadores.

2.14. Fora do Escopo

  • Integração Direta com Gateways de Pagamento: O sistema está preparado para registrar pagamentos, mas a automação via integração com serviços como Stripe ou PagSeguro não foi implementada.
  • Módulo de Treinos e Avaliação Física: Funcionalidades para prescrição de treinos e acompanhamento da evolução física dos alunos.

2.15. Protótipo de Interface (Descrição)

  • Tela de Login: Design limpo com campos para e-mail e senha, botão "Entrar" e link para recuperação de senha.
  • Página Principal (Dashboard): Apresenta um painel dinâmico com cards de métricas principais (clientes ativos, receita mensal, acessos do dia), gráficos de desempenho e listas de acesso rápido (mensalidades a vencer, produtos com baixo estoque).
  • Menu de Navegação: Menu lateral colapsável com ícones e texto, dando acesso a todas as seções principais do sistema (Dashboard, Clientes, Produtos, Categorias, Vendas, Reconhecimento, Academias, Planos, Usuários, Cadastro de Usuário).

  • Formulários de Cadastro/Edição: Utilizam componentes de formulário consistentes, com validação de dados em tempo real e feedback claro para o usuário.
  • Modo Kiosk (Autorização de Entrada): Interface de tela cheia com um feed de vídeo da câmera em destaque, sobreposições visuais para indicar a detecção de rosto e mensagens de status claras ("Acesso Liberado", "Acesso Negado").

4.0 DESCRIÇÕES DE CASO DE USO

4.1 Módulo de Autenticação (UC01)

Caso de Uso: Fazer Login

  • Ator Principal: Usuário (Funcionário/Administrador).
  • Descrição: Permite que o usuário se autentique no sistema "JyM" informando suas credenciais (usuário e senha) para obter acesso às funcionalidades conforme seu nível de permissão.
  • Pré-condição: O usuário deve possuir cadastro ativo no sistema.
  • Pós-condição: Sessão autenticada criada, usuário redirecionado ao dashboard com contexto de academia definido.

4.7 Módulo de Controle de Acesso - Kiosk (UC07)

  • Ator Principal: Cliente.
  • Ator Secundário: Sistema Kiosk.
  • Descrição: Permite que o cliente acesse a academia posicionando o rosto na câmera do kiosk para autenticação via reconhecimento facial.

  • 8.7. O sistema dispara evento DashboardUpdated via WebSocket.
  • 8.8. O kiosk exibe "BEM-VINDO, {nome}!" com foto do cliente.
  • Fluxo Alternativo:
  • 8.4a. Rosto não reconhecido: Kiosk exibe "Rosto não identificado. Use CPF/Código" e ativa fluxo alternativo (UC07-b).
  • 8.5a. Status = "Inadimplente": Kiosk exibe "ACESSO BLOQUEADO - Mensalidade em atraso. Procure a recepção."
  • 8.5b. Status = "Inativo": Kiosk exibe "CADASTRO INATIVO".
  • Relações:
  • <<include>>: Extrair Descriptor, Comparar Descritores, Verificar Status, Registrar Entrada.
  • <<extend>>: Autorizar Entrada por CPF/Código.
  • Rate Limiting: 30 req/min por IP.

Descrição do Diagrama de Sequência: Autorizar Entrada por Reconhecimento Facial

O diagrama de sequência "Autorizar Entrada" modela o fluxo completo de autenticação de clientes no kiosk via reconhecimento facial, incluindo verificação de status financeiro e comunicação em tempo real.

O processo inicia com o :Cliente posicionando o rosto na câmera do :Kiosk. O :Kiosk captura frame de vídeo, processa com face-api.js (modelos: tinyFaceDetector, faceLandmark68Net, faceRecognitionNet) e extrai descriptor facial (array de 128 floats). O descriptor é enviado via POST para /api/kiosk/authenticate com rate limiting (30 req/min).

O :FaceRecognitionController:

  1. Carrega todos os :FaceDescriptor da academia ativa
  2. Calcula distância euclidiana entre descriptor recebido e cada descriptor armazenado
  3. Identifica match com menor distância (threshold < 0.6)

Fragmento alt baseado no resultado:

  • Rosto não reconhecido (distância >= 0.6): Retorna match: false, kiosk exibe "Rosto não identificado" e oferece entrada via CPF/Código
  • Rosto reconhecido: Carrega :Cliente vinculado e verifica status

Sub-fragmento alt para status do cliente:

  • Status = "Ativo": :EntradaService cria registro em :Entrada (método: "Reconhecimento Facial"), dispara :DashboardUpdated, retorna { success: true, cliente: {...} }. Kiosk exibe "BEM-VINDO, {nome}!" com foto
  • Status = "Inadimplente": Retorna { success: false, message: "Acesso bloqueado - Mensalidade em atraso" }. Kiosk exibe alerta vermelho
  • Status = "Inativo": Retorna { success: false, message: "Cadastro inativo" }

O diagrama destaca a integração entre face-api.js, Laravel API e WebSocket para atualização do dashboard em tempo real.

Receber Mercadoria (Compra) com Rateio de Custos

Descrição do Diagrama de Sequência: Receber Mercadoria

O diagrama de sequência "Receber Mercadoria" modela o processo de confirmação de recebimento de compras, incluindo rateio proporcional de custos (frete, impostos) e recálculo de custo médio ponderado.

Pré-requisito: Compra já criada com status "Aberta" (itens, fornecedor, valores definidos).

O :Funcionário acessa detalhes da compra (/compras/{id}) e clica "Receber Mercadoria". O :CompraController:

  1. Carrega :Compra com relacionamentos (itens, produtos)
  2. Calcula valor total dos itens (soma de quantidade × preço unitário)
  3. Rateia frete proporcionalmente: freteItem = (valorItem / valorTotal) × freteTotalCompra
  4. Rateia impostos: impostoItem = (valorItem / valorTotal) × impostosTotalCompra

Para cada :ItemCompra:

  1. Calcula custo rateado: custoTotal = (quantidade × precoUnitario) + freteRateado + impostoRateado
  2. Calcula custo unitário final: custoUnitario = custoTotal / quantidade
  3. Consulta :Produto para obter estoque atual e custo médio atual
  4. Recalcula custo médio ponderado:

novoCustoMedio = ((estoqueAtual × custoMedioAtual) + (quantidade × custoUnitario)) / (estoqueAtual + quantidade)

  1. Atualiza :Produto com novo estoque e novo custo médio
  2. Cria :MovimentacaoEstoque (tipo: "entrada", origem: "compra", custoUnitario)

Após processar todos os itens:

  1. Atualiza :Compra com status "Recebida" e dataRecebimento
  2. Cria :ContaPagar (fornecedor, valor total, vencimento, origem: "compra")
  3. Registra em :ActivityLog
  4. Retorna compraRecebida: true

Fragmento alt: Compra já recebida: erro: "Compra já foi recebida anteriormente".

O diagrama destaca o modelo de custeio por absorção, onde frete e impostos são incorporados ao custo do produto para correta margem de lucro.

8.0 CONCLUSÃO

O desenvolvimento do sistema "JyM" foi conduzido com base nas melhores práticas da engenharia de software, utilizando a Unified Modeling Language (UML) para modelar e documentar suas funcionalidades, estrutura e comportamento. A elaboração dos casos de uso, diagramas de classes, diagramas de sequência e o modelo de banco de dados permitiu uma abordagem sistemática e estruturada, garantindo que os requisitos funcionais fossem capturados, validados e implementados de forma consistente. Os casos de uso detalharam as interações dos atores ("Funcionário/Administrador" e "Cliente") com o sistema, enquanto o diagrama de classes definiu a estrutura estática, mapeando as entidades principais (Usuário, Cliente, Mensalidade, Entrada) e seus relacionamentos. Os diagramas de sequência complementam a análise, especificando o comportamento dinâmico para cenários como "Fazer Login", "Cadastrar Cliente" e "Gerenciar Mensalidades". Por fim, o modelo de banco de dados relacional foi projetado e testado, assegurando a persistência eficiente dos dados com integridade e escalabilidade.

Perspectivas de Evolução do Sistema

  • Controle de presenças em aulas ou treinos personalizados, permitindo que o sistema registre a participação dos clientes em horários e modalidades específicas;
  • Relatórios gerenciais avançados, com gráficos de desempenho financeiro, evolução de alunos e métricas de engajamento, apoiando a tomada de decisões estratégicas;
